## 2026-06-15 — `MicSource`: mlx-whisper promoted to real deps; `Transcriber` seam; segment→Utterance policy (T-104)

**Decided by:** Claude Code (sensing-engineer) — T-104
**Status:** accepted
**Context:** T-104 is the last seam in the Phase-1 audio path: wire the T-102 `AudioSource` + T-103 `SileroVad` + the T-101-chosen ASR runtime (`mlx-whisper base.en`) into `Utterance` events behind the **frozen `TranscriptSource` seam**, so `MicSource` replaces `ScriptedSource` in `AttentionLayer` with no change to the core. Three calls: (1) where the ASR dependency lives now that it's shipped runtime; (2) how to keep the segment→`Utterance` logic testable without the real model; (3) what audio a "speech segment" is and where `Utterance.ts` comes from.
**Decision:** (1) **Promote `mlx-whisper` from the isolated `asr-spike` uv group into real `[project.dependencies]`** — it is now a shipped always-on runtime dependency, same reasoning as `sounddevice`/`silero-vad` in T-102/T-103. `pywhispercpp` stays in the `asr-spike` group as the documented fallback/benchmark dep only (not shipped). (2) Put ASR behind a tiny **`Transcriber` Protocol** — `transcribe(waveform: np.ndarray, sample_rate: int) -> str` — the audio-path analogue of `SummarizerBackend`/`WallBackend`/`FrameClassifier`. Default `MlxWhisperTranscriber` runs `mlx_whisper.transcribe(waveform, path_or_hf_repo="mlx-community/whisper-base.en-mlx")`, **imported lazily** so importing the module never loads MLX; tests inject a `FakeTranscriber`. (3) A **speech segment** is the VAD's debounced `on_speech_start`→`on_speech_end` window: `MicSource` accumulates the frames between those edges, concatenates them into one float32 waveform, and transcribes once at the `speech_end` edge. **`Utterance.ts` is stamped from the VAD timeline** (`frames_seen × frame_samples / sample_rate` — the audio position of the segment's end), **never a wall clock**, preserving the no-hidden-clock invariant all the way out to the live source. Empty/whitespace ASR text is dropped (no empty utterances); an open segment at end-of-stream is flushed. Speaker is a fixed placeholder (`DEFAULT_SPEAKER`) — diarization is out of scope for v0. `MicSource` drives the **same** `TurnTakingGate` the `SummonController` reads (via the VAD edges), so summon/interjection timing runs on live audio unchanged.
**Rationale:** The `Transcriber` seam is what lets the entire segment→`Utterance` logic + gate-edge wiring be unit-tested deterministically with **no mic and no model** (a `FakeTranscriber` + `FakeAudioSource` + `EnergyFrameClassifier`), exactly mirroring how every other heavy boundary in the project is faked. Deriving `ts` from the frame count (not `time.monotonic()`) keeps the live source consistent with the gate's injected clock and honors the project-wide "no hidden clock" constraint — the `RollingWindow` evicts the live transcript by the same timeline the gate gaps on. Transcribing the whole segment once (rather than streaming partials) matches the T-101 finding that a VAD-segmented utterance is the natural ASR unit and is tens of ms on this M5 — far inside the ~2 s budget — so no streaming complexity is needed for v0. Promoting `mlx-whisper` to real deps (and demoting it out of the spike group) keeps the dependency surface honest: the spike group now lists only the unshipped fallback.
**Alternatives considered:** Streaming/chunked ASR within a segment (rejected for v0 — unnecessary at this latency; the whole-segment unit is simpler and was what T-101 validated). Stamping `ts` from `time.monotonic()` at segment close (rejected — reintroduces the hidden wall clock the whole architecture forbids; the frame timeline is exact and deterministic). Keeping `mlx-whisper` in the `asr-spike` group and importing from there (rejected — it's a shipped runtime now, so it belongs in real deps per the T-101 dep policy, which explicitly said T-104 promotes the chosen runtime). Calling `mlx_whisper` on a WAV file path (rejected — it accepts a float32 waveform directly, so no temp-file round-trip; the `AudioSource` already yields float32). Diarization to fill `speaker` (rejected — out of scope for the single-user v0; a placeholder label suffices and a real diarizer can fill the same field later).

## 2026-06-15 — Silero VAD via `silero-vad` (torch) behind a `FrameClassifier` seam (T-103)

**Decided by:** Claude Code (sensing-engineer) — T-103
**Status:** accepted
**Context:** The approved stack names **Silero VAD** for voice-activity detection (WebRTC VAD is the listed pre-swap alternative). T-103 integrates it to segment the `AudioSource` (T-102) frame stream into speech/silence and drive the frozen `TurnTakingGate` edge API. Two calls: which Silero package, and how to keep the gate-driving *logic* testable without loading a torch model in the unit tests.
**Decision:** Use the official **`silero-vad`** pip package (pulls **torch** + `torchaudio`), recorded as a real package dep (torch was already present from the ASR/Phase-2 MLX stack, so this adds `silero-vad`+`torchaudio`, not a fresh torch). Wrap it in **`SileroVad`** (`audio/vad.py`), which debounces a per-frame speech decision into clean **speech-start/speech-end edges** and delivers them to an injected `TurnTakingGate` — **edges only, no timestamps** (the gate stamps from its own clock, per the T-006 decision; this VAD aligns to that seam, does not reshape it). The per-frame "is this speech?" decision is the one model-dependent part, so it sits behind an injected **`FrameClassifier`** Protocol: default `SileroFrameClassifier` (loads `load_silero_vad()` **lazily**, scores each 512-sample/16 kHz frame, speech iff prob ≥ threshold); tests inject `EnergyFrameClassifier` (pure RMS ≥ threshold). Hysteresis (`speech_start_frames`, `silence_end_frames`) + `threshold` are constructor-injected for Phase-5 calibration.
**Rationale:** `silero-vad` is the maintained first-party package for the approved VAD; its `VADIterator` confirms the same start/end edge model I emit (verified on synthetic tone frames). The **`FrameClassifier` seam is the audio-path analogue of the core's injected-backend discipline** (`SummarizerBackend`/`WallBackend`): it isolates the heavy torch model so the **edge-sequencing logic that actually drives the gate** is unit-tested deterministically with no torch, no model download, no mic — while the real model is still exercised by an optional live-mic test (which **ran and passed** here, permission granted). Measuring the VAD's own timing in **frames** (not a clock) keeps the entire audio path clock-free, preserving the "gate is the single clock owner" invariant. The VAD-side endpoint hangover (~200 ms) is kept far shorter than the gate's ~2 s politeness gap so acoustic vs. social timing stay separated. **Dep weight noted:** torch is heavy (~hundreds of MB) but already in the always-on stack via MLX/ASR; `silero-vad` itself is tiny (~9 MB model).
**Alternatives considered:** WebRTC VAD / `webrtcvad` (the listed alternative — rejected: less accurate than Silero, and Silero is the approved choice; revisit only if torch becomes a liability). Silero's `VADIterator` directly as the segmenter (rejected for the seam — it bundles model + debounce together, so the debounce logic couldn't be tested without torch; re-implementing the frame-debounce around an injectable classifier makes the gate-driving logic testable and keeps the threshold/hysteresis in one calibration place). Putting `silero-vad` in a spike group (rejected — this is the shipped always-on VAD, belongs in real deps, same reasoning as `sounddevice` in T-102).

## 2026-06-15 — Mic capture: `sounddevice`/PortAudio + the `AudioSource` abstraction (T-102)

**Decided by:** Claude Code (sensing-engineer) — T-102
**Status:** accepted
**Context:** Phase 1 needs an always-on microphone capture loop feeding the VAD (T-103) and ASR (T-104). Two calls: (1) the capture library, and (2) how to keep the VAD and the whole test suite from depending on real hardware. The approved stack names Silero VAD and local ASR but left the *capture* library unspecified.
**Decision:** Capture audio with **`sounddevice`** (a PortAudio binding) at **16 kHz mono float32, 512-sample/32 ms frames** — Silero VAD's native geometry, so no resampling/reblocking between capture and VAD. Introduce a small **`AudioSource`** abstraction (`src/jarvis/audio/source.py`): a `Protocol` yielding fixed-size `AudioFrame` chunks, with a real `SoundDeviceMicSource` (`audio/mic.py`) and a hardware-free `FakeAudioSource` behind it. The PortAudio real-time callback pushes frames into a **bounded `RingBuffer`** (overwrites oldest + counts `overflows` when full — bounded memory for an always-on process, no unbounded growth); the consumer pops on its own schedule. `sounddevice` is imported **lazily** inside `start()` so importing the package never requires a working PortAudio. Permission/no-device failures surface as typed errors (`MicPermissionError`/`NoInputDeviceError`) — **never** papered over with synthetic audio. `sounddevice` + `numpy` go into the real `[project.dependencies]` (this is the actual always-on runtime now), not a spike group.
**Rationale:** `sounddevice` is the de-facto, well-maintained Python PortAudio binding, gives a real-time callback (the right model for never-drop capture), and installs cleanly on macOS with PortAudio bundled — no Homebrew step. The `AudioSource` seam is the audio-path analogue of the core's injected-backend discipline: it keeps the VAD/ASR and every test off real hardware (driven by `FakeAudioSource` synthetic frames), exactly mirroring how `ScriptedSource` keeps the core off a real mic. The bounded ring buffer is the standard real-time-audio hand-off and is what makes "always-on" memory-safe. Lazy import keeps test collection green on any machine. **Live smoke test ran on this M5** (permission already granted): ~1.47 s captured, 46 frames, 0 overflows — real capture, documented honestly per the no-fabrication rule.
**Alternatives considered:** `pyaudio` (rejected — older, needs a separate PortAudio install, clunkier callback API). Raw `ffmpeg`/`avfoundation` subprocess capture (rejected — heavier, harder to frame-align to Silero's window, more failure surface). Putting `sounddevice` in a spike group like the ASR deps (rejected — unlike the ASR benchmark, this IS the shipped always-on capture path, so it belongs in real deps). Resampling from a 44.1/48 kHz device default (rejected for now — request 16 kHz directly from PortAudio; revisit only if a device refuses the rate).

## 2026-06-15 — ASR runtime: mlx-whisper (base.en) over whisper.cpp (T-101 spike)

**Decided by:** Claude Code (sensing-engineer) — T-101 spike
**Status:** accepted
**Context:** The approved stack deferred the local ASR runtime to a Phase-1 spike — mlx-whisper vs whisper.cpp — to be settled empirically on the actual M5 rather than guessed. T-101 benchmarked both on this machine (Apple M5 Pro, 64 GB) at the `base.en` model: latency/RTF, WER on a known synthesized reference clip, isolated memory, and a 40× single-session sustained-load read. Full methodology + numbers in `docs/audio/asr-spike.md`.
**Decision:** Use **`mlx-whisper` at `base.en`** as the Phase-1 ASR runtime (English-only; the single English-speaking developer is the v0 user), with **`small.en` as the documented upgrade lever** and **whisper.cpp (`pywhispercpp`) as the documented fallback**. The runtime sits behind the frozen `TranscriptSource` seam, feeding `MicSource` (T-104).
**Rationale:** Latency and accuracy were effectively **tied** at `base.en` — both runtimes are ~25–125× faster than real time (short ~3.8 s utterance: mlx 73 ms / whisper.cpp 52 ms; both negligible against the ~2 s offer-to-help budget) and both scored 0.0 % WER on clean short speech / 1.7 % on a 17 s paragraph. The deciding factor is **runtime strategy**: mlx-whisper runs on **MLX/Metal/unified-memory — the same accelerator stack Qwen2.5 will use in Phase 2** — so the always-on ambient half standardizes on **one** local-inference stack to budget and profile, rather than MLX plus a second ggml/Metal runtime. mlx is also maintained Apple-Silicon-first (whisper.cpp's ggml build on this M5 couldn't use the newest tensor API and fell back). `base.en` is chosen to **leave the most M5 headroom for the SLM** (the real always-on cost center). whisper.cpp was marginally faster and leaner (326 MB vs 463 MB RSS, no torch dep) — kept as the fallback, but ~15 ms doesn't justify a second inference runtime.
**Alternatives considered:** whisper.cpp/`pywhispercpp` as primary (rejected as primary, kept as fallback — faster/leaner but a separate runtime from the MLX SLM; the speed edge is irrelevant at this budget). `tiny.en` (rejected — saves negligible latency for an accuracy cost). `small.en` now (deferred — only move up if captured-audio WER demands it *and* the combined ASR+SLM budget still clears). The other approved-stack alternatives (`faster-whisper`, Moonshine, Parakeet) were not benchmarked — out of scope for the two-candidate spike; revisit only if both candidates fail in the wild. **⚠️ Open coexistence item:** this spike measured ASR in isolation; the **ASR + Qwen2.5 concurrent always-on budget** must be measured jointly with local-ml-engineer before model sizes freeze on either side (flagged in `asr-spike.md`).

Recorded here per the dependency policy.

A throwaway venv outside uv (rejected — not reproducible from the project, fights the uv-managed-toolchain decision). The `dev` group (rejected — these aren't dev tooling like pytest/ruff; a purpose-named group documents intent).

## 2026-06-16 — ScriptedSource carries inter-line timing and drives the gate + clock (T-008)

**Decided by:** Claude Code (core-engineer) — T-008
**Status:** accepted
**Context:** The orchestrator's Path B (interjection) only fires once the `TurnTakingGate`'s ~2 s politeness gap has elapsed, and the gate measures silence off the single injected clock (the "no hidden clock" constraint). So a canned conversation that is merely a list of `(speaker, text)` lines can *never* produce an interjection in the end-to-end mock run: no time passes between lines, the gap never opens, and the acceptance criterion ("at least one correct interjection") is unreachable. The reference prototype dodged this with a real `time.sleep(pace)` and an internal `time.monotonic()` — both barred in the real package.
**Decision:** The `TranscriptSource` seam stays minimal (`utterances() -> Iterable[Utterance]`), but the Phase-0 `ScriptedSource` fill carries **inter-line timing**: each line is a `ScriptedLine(speaker, text, gap)` where `gap` is the seconds of silence following the line. As the source plays a line it (1) fires the shared `TurnTakingGate`'s `on_speech_start`/`on_speech_end` edge events — the exact edges a real VAD/`MicSource` emits — (2) advances the **injected** clock (`clock_advance`, e.g. `SimulatedClock.advance`) by the modeled speech duration then by the `gap`, and (3) stamps `Utterance.ts` from that same clock. The orchestrator and the source share one gate + one clock, so when `SummonController.consider_interjection` reads the gate predicates they reflect the conversation's pacing. The orchestrator's `run_scripted` classmethod wires this in one call. No `time.sleep`, no internal clock — the whole mock pipeline is deterministic with zero real time elapsed.
**Rationale:** Keeps the seam the live `MicSource` implements tiny and timing-free, while giving the Phase-0 source exactly the two things a real audio path provides — VAD speech boundaries and a clock — so the *same* gate/controller logic is exercised end-to-end in mock mode that will run live. Driving the injected clock (not sleeping) makes the demo instant and the acceptance tests deterministic, and honors the project-wide injected-clock constraint down into the test/demo driver. The `MicSource` swap (Phase 1, T-104) replaces `ScriptedSource` without the orchestrator or the gate changing: `MicSource` feeds the real VAD edges and a real `time.monotonic` clock through the same wiring.
**Alternatives considered:** (a) Real `time.sleep(pace)` between lines like the prototype (rejected — non-deterministic, slow tests, and it needs a real wall clock the core is forbidden to read). (b) Put a `ts`/`gap` on `Utterance` and have the orchestrator advance time (rejected — the orchestrator shouldn't own pacing or drive the clock; the *source* is the timeline, mirroring how `MicSource` is the source of VAD timing). (c) Have the orchestrator synthesize gate events from utterance timestamps (rejected — couples the orchestrator to inferring speech boundaries it shouldn't guess; the source already knows them, and a real VAD emits them directly). Relatedly, the manual clock was moved from `tests/clock.py` into the package as `jarvis.clock.ManualClock` so the runnable demo doesn't import from `tests/`; `tests/clock.py` re-exports it as `SimulatedClock` (one implementation, the harness name unchanged).

## 2026-06-15 — SummonController emits a decision, the orchestrator assembles the handoff (T-007)

**Decided by:** Claude Code (core-engineer) — T-007
**Status:** accepted
**Context:** The module map said `SummonController` "turns gate + detector signals into either an `EngagementHandoff` (summon) or an `Interjection` offer." But the full `EngagementHandoff` needs `summary` (`LivingSummary.text`) and `recent_excerpt` (rendered `RollingWindow` lines) — neither of which the controller holds; both live in the orchestrator (`AttentionLayer`, T-008). Letting the controller build the handoff would force it to take the summary + window (or the whole orchestrator) as dependencies, fattening the success-metric-critical state machine and coupling it to plumbing it doesn't otherwise touch.
**Decision:** `SummonController` is a **pure decision machine**. It emits a frozen `SummonDecision` — `reason: TriggerReason` (`SUMMON` | `INTERJECTION`), an optional `Interjection` (Path B only), and `detail` (the summon text, Path A) — plus a `handoff_reason()` helper returning the `"summon"` / `"wall:<category>"` wire string. The **orchestrator** (T-008) turns a summon decision into an `EngagementHandoff` (adding `summary` + `recent_excerpt`) and a interjection decision into a dispatched offer. The controller's only inputs are the injected `TurnTakingGate` and a per-call `WallVerdict`; its only carried state is the back-off signature. `Interjection`, `SummonDecision`, `TriggerReason` land in `types.py` at T-007; `EngagementHandoff`'s shape is frozen there too but it is *built* at T-008.
**Rationale:** Keeps the riskiest module (qa-tuning's mandatory-review target, the success metric) tiny and fully unit-testable in isolation over (gate, verdict) — no summary/window fakes needed to test the decision logic. Handoff assembly stays in the one place that already owns the summary and window, so there's a single assembly site and no duplicated context-gathering. The seam to voice-integration-engineer (`EngagementHandoff`) is unchanged — its shape is still frozen here.
**Alternatives considered:** (a) Controller builds the full `EngagementHandoff` (rejected — needs the summary + window injected, coupling the state machine to orchestrator-owned context and bloating the review surface). (b) Controller returns a raw `WallVerdict` / bare bool and the orchestrator re-derives the path (rejected — loses the typed two-path decision and the `handoff_reason` convention; pushes Path-A-vs-B branching into the caller). (c) The controller also runs the `WallDetector` internally (rejected — the orchestrator already gates *when* to run detection on a cheap wall signal per the event flow; passing the verdict in keeps the controller a pure function of its two real inputs and avoids it deciding when to spend the detector).

## 2026-06-15 — TurnTakingGate event-input API: VAD boundary events on the injected clock (T-006)

**Decided by:** Claude Code (core-engineer) — T-006
**Status:** accepted
**Context:** The module map froze `TurnTakingGate`'s three *output* predicates (`settled` / `politeness_gap_elapsed` / `speech_resumed`) but left the *input* side undesigned — how speech/silence boundaries are fed in — and qa-tuning (T-009) flagged it as the open interface gap to close in T-006. The clock side was already settled (`now: Callable[[], float]`).
**Decision:** The gate consumes two **edge events** off the VAD timeline — `on_speech_start()` and `on_speech_end()` — and nothing else. The events carry **no timestamp argument**; the gate stamps them from the injected `now()` at delivery, keeping one clock source of truth. Silence is measured from the **most recent** `on_speech_end()`. `on_speech_start()` re-arms the gate (silence predicates fall back to `False`) and latches `speech_resumed()` `True` if it interrupted an already-open gap; the next `on_speech_end()` clears that latch. The three predicates are **pure reads** of (state, now) — idempotent, no consume-on-read. The two thresholds (`settle_seconds` ≈ 0.6, `politeness_gap_seconds` ≈ 2.0, the latter must be ≥ the former) are constructor-injected.
**Rationale:** The gate reasons about *durations of silence*, so the two things it needs are the two transition instants — not a per-frame level stream. An edge API is smaller, has no threshold-crossing/debounce bookkeeping inside the gate, and maps 1:1 onto Silero VAD's segment callbacks in Phase 3 (a VAD segmenter emits exactly speech-start / speech-end). Timestamping from the single injected clock (rather than a `ts` arg per event) preserves the "no hidden clock / one clock convention" constraint and lets `SimulatedClock` drive every transition deterministically. Pure-read predicates let `SummonController` and tests poll freely.
**Alternatives considered:** A per-frame `feed(is_speech: bool)` / `on_vad(level)` poll (rejected — pushes endpoint debouncing and frame-rate coupling into the gate; the gate would have to detect edges itself). A single `feed(event)` with an event enum (rejected — two named methods are clearer at call sites and for the VAD adapter to target; no real expressiveness lost). Passing an explicit `ts` with each event (rejected — a second clock source that could drift from `now()`; the gate already has the injected clock). Making the predicates consume-on-read / event-emitting (rejected — the module map's predicates are *queries*; `SummonController` needs to poll them across ticks without side effects).

## 2026-06-15 — Test harness: shared simulated clock + seam fakes (T-009)

**Decided by:** Claude Code (qa-tuning) — T-009
**Status:** accepted
**Context:** The six core modules (T-002…T-008) all need a deterministic time source and doubles for the swappable backend/adapter seams. Letting each test task reinvent these would drift — different clocks, different fake shapes — and risk tests pinned to internals that break when the real model/audio/voice backends swap in (Phases 1, 2, 4).
**Decision:** One shared harness under `tests/`: `clock.py` (`SimulatedClock` — injected, monotonic-by-construction, driven with `advance`/`set`, no real `sleep`), `fakes.py` (`FakeSummarizer`, `FakeWallBackend`, `FakeResponder`, `FakeVoice` — each presets a return and records calls; plus a `WallVerdictLike` stand-in until T-005 freezes the real `WallVerdict`), `conftest.py` (fixtures), and `test_harness.py` (self-tests). The governing rule for all core-module tests: **assert on external behavior — return values, emitted events, seam calls — never on private fields.** Modules receive time as `now: Callable[[], float]` (pass `clock.now`) and seams via the constructor. Conventions documented in `docs/qa/eval-plan.md`.
**Rationale:** Bind the test infra to the *work* (the seam contract in `module-map.md`), not to each task, so all six modules share one vocabulary and behavior-pinned tests survive the mock→real backend swaps the architecture is built to allow. The injected clock is the mechanism the module map's "no hidden `time.monotonic()`" constraint exists to enable.
**Alternatives considered:** Per-task ad-hoc clocks/fakes (rejected — drift, duplication, no shared conventions). `freezegun`/`unittest.mock` time patching (rejected — patches a hidden global clock, which fights the injected-clock constraint and the "no I/O/hidden state in core" design; explicit injection is more honest and deterministic). `unittest.mock.Mock` for the seams (rejected for the default path — hand-written fakes give readable preset/record APIs and typed verdict helpers; `Mock` is available to tests that want it).

## 2026-06-15 — Phase 0 toolchain: uv + src-layout + pytest + ruff

**Decided by:** Claude Code (core-engineer) — T-001 scaffold
**Status:** accepted
**Context:** T-001 stands up the real `jarvis` package. The stack requires Python 3.11+, but the machine's system `python3` is 3.9.6 and no 3.11+ was installed (no pyenv, no brewed python3.11).
**Decision:** Use **uv** (0.11.x, installed via the official standalone installer to `~/.local/bin` — no sudo, no interactive prompts) to manage a pinned **CPython 3.11.15** and the venv. Package uses a **src-layout** (`src/jarvis/`) with a `pyproject.toml` (`requires-python = ">=3.11"`, hatchling build backend). **pytest** is the test runner (`pythonpath = ["src"]`, `testpaths = ["tests"]`); **ruff** is lint + format (line-length 100, target py311, rule set E/W/F/I/N/UP/B/C4/SIM). `prototypes/` is excluded from ruff — it is reference, not the package. `uv.lock` and `.python-version` are committed to pin the toolchain.
**Rationale:** uv is the fastest, fully non-interactive way to obtain a managed 3.11 without touching the system Python or running a heavy/interactive installer — exactly the environment-note guidance in the task. src-layout prevents accidental imports of the un-built package and keeps the import surface honest. ruff is one tool for lint + format, fast, zero-config-friendly.
**Alternatives considered:** pyenv (rejected — not installed; heavier setup than the uv standalone binary). Homebrew `python3.11` + stdlib `venv` (viable fallback but slower to provision and no lockfile). flat-layout (rejected — src-layout is the project's recommended layout and avoids shadowing). black + flake8 + isort (rejected — ruff subsumes all three in one fast tool).

## 2026-06-15 — Asymmetric dual-summon (fast when called, polite when interjecting)

**Decided by:** discussion (human + research)
**Status:** accepted
**Context:** Jarvis can engage two ways — a wake-word summon and a proactive interjection. Turn-taking/endpointing research (LiveKit VAD+semantic endpointing; RESPOND's turn-claim aggressiveness) shows the agent must distinguish a completion pause from a thinking pause, and that an *uninvited* third party should behave differently from an *addressed* assistant.
**Decision:** Two initiation paths with deliberately opposite timing. Path A (summon) responds fast (~500–700 ms endpoint, low confidence bar). Path B (interjection) hangs back (~2 s politeness gap, confidence ≥ ~0.70, aborts on resumed speech). Completeness/intent is folded into the WallDetector call rather than a separate end-of-turn model. Defaults are conservative and tuned later on captured conversations.
**Rationale:** A false summon is harmless; a false interjection is the assistant talking over people. The asymmetry encodes that social cost directly.
**Alternatives considered:** A single uniform endpoint threshold for both paths (rejected — would make interjection either too eager or summon too slow). A full-duplex end-to-end model like Moshi (rejected for v0 — heavy, not debuggable; documented as future direction).

## 2026-06-15 — Cloud only at the moment of answering; ambient stays local

**Decided by:** human

**Decision:** The ambient half (mic → VAD → ASR → summary → wall detection) runs 100% on-device. The cloud (Claude for the answer, ElevenLabs for the voice) is invoked only once Jarvis is actually engaged. The rolling transcript is ephemeral by default.

## 2026-06-15 — Compliance posture: none (privacy via hard-nos)

**Decided by:** human
**Status:** accepted
**Context:** The transcript (utterance / rolling-window / living-summary) carries personal/sensitive spoken content, so the schema flags PII. The PDR's PII-without-compliance check was raised explicitly during grill.
**Decision:** No formal compliance regime applies — this is a personal, single-user, on-device, ephemeral tool. The privacy posture is enforced through the hard-nos (no ambient audio to cloud, no persistence by default) rather than a regulatory framework.
